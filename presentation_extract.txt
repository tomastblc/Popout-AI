Primeiro Capítulo: Estrutura do Tabuleiro 
Importância da Performance 
A estrutura do tabuleiro é crucial para a 
performance, especialmente com o 
algoritmo MCTS . Este método exige um 
volume elevado de simulações para cada 
jogada, tornando a eficiência de dados um 
requisito fundamental. Implementação Bitboard 
Utilizamos Bitboards  (inteiros de 64 bits) 
para representar o estado do jogo. 
● Operações Dr op  e P op  são efetuadas 
via bit -shif ting  aritmético. 
● Redução de 95% no tempo de 
execução. 
● De 20s para apenas 0,9s.

Variantes de MCTS 
Exploração via C 
C baixo:  Afunila a exploração em jogadas com alta 
taxa de vitória. 
C elevado:  Prioriza a exploração de nós menos 
visitados. Expansão & Rollout 
Expansão Aleatória:  Escolhe k jogadas aleatórias em 
vez de todas. 
Heuristic Rollout:  Simulação guiada por regras 
específicas em vez de aleatória. 
Outras Otimizações 
Epsilon Greedy:  Rollout inteligente com probabilidade de movimentos aleatórios. 
Depth Limited:  Interrompe a simulação ao atingir um limite de profundidade. 
Variantes obtidas ao aplicar mudanças ao algoritmo-base.

Gerar os datasets 
Processo de Geração 
Para gerar os datasets a serem treinados criamos um script que coloca a versão base do 
MCTS a jogar contra si próprio (durante 750 jogos ) que, a cada passo memoriza o 
tabuleiro e a ação que escolheu registando os mesmos num ficheiro CSV . 

--- PAGE 7 ---
Dataset do Torneio 
Visão Geral do Dataset 
Para gerar o dataset do torneio entre os sete agentes do MCTS  a ideia geral é a mesma, 
sendo a grande diferença o facto de cada par jogar 2 vezes  entre si, resultando num 
dataset bastante rico em informação aprendendo várias estratégias de jogo. 

--- PAGE 8 ---
Dataset do Torneio Perturbado 
Processo de Geração e Recuperação 
Este processo pega no dataset do torneio anterior e realiza os seguintes passos: 
● Lê linha a linha o ficheiro CSV e reconstrói o tabuleiro. 
● Faz entre 2 a 6 jogadas aleatórias e descarta estados terminais. 
● Três melhores agentes MCTS definem a jogada de recuperação após o baralhamento. 
● Apaga as colunas desnecessárias para não confundir o ID3

Interface 
A interface é uma das partes mais importantes do jogo. 
No arranque o script carrega o motor do jogo, a IA de pesquisa em tempo real MCTS e a árvore 
de decisão treinada. 
Game loop: 
● Desenha o estado atual do tabuleiro no terminal 
● Chama as funções de verificação de vencedor ou se é possível declarar um empate 
● Se for esse o caso, pede o input ao jogador humano 
● Aplica a jogada com uma função própria para esse efeito 

--- PAGE 15 ---
Interface Cont. 
Tratamento do input: 
A função get_human_move()  divide a string (ex: drop / 3, verifica a validade do número e cruza com a 
função legal_moves() .
MCTS vs ID3 
● MCTS  Instanciado a cada jogada, analisando o estado ao vivo e simulando em tempo real. 
● ID3  Converte o tabuleiro para dicionário e processa via função classificar() .
● Fallback:  Se a árvore encontrar uma situação não treinada, executa o primeiro movimento legal.

Com este projeto conseguimos concluir que uma boa exploração dos

para a descoberta do melhor agente possível.
